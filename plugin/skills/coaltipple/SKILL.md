# CoalTipple — routing contract

> [!WARNING]
> **PLATFORM GATE -- actuates on Claude Code ONLY.** CoalTipple routes by spawning a worker at a CHOSEN model + effort, which needs a platform whose spawn tool accepts a worker **model** parameter. **Confirmed working: Claude Code** (the `Agent`/`Task` tool takes `model`). **Confirmed BROKEN: Antigravity** -- `invoke_subagent`/`define_subagent` expose **no model parameter**, and AG has **no separate effort knob** (low/mid/high are baked into model *names*), so a spawned worker silently runs the **parent's** model. On any platform without an agent-selectable worker model, **do NOT pretend to delegate-down or escalate-up** -- the spawn cannot change tier and routing is a silent no-op. Not on Claude Code -> this skill does not apply. (Platform support reviewed monthly.)

> **VERSION GATE -- validated on Claude Code 2.1.143.** On a different CC version, rebuild the ranking on first route and verify routing before relying on it (it degrades safe -- unknown model -> strong tier, failed spawn -> fall -- but the self-heal is unverified outside 2.1.143). This gate lifts once newer Claude Code versions are verified.

You are **main** (depth-0). You decide routing + spawn everything yourself. **Workers are leaves BY POLICY — routing stays depth-0: give a worker a bounded task-contract (its done-criteria make it RETURN, never open-endedly spawn its own workers), and a worker that fails RETURNS its result for main to re-route. This holds whether or not the platform caps subagent nesting — nesting varies by CC version (capped before 2.1.172, ON from 2.1.172), so never depend on the cap: robustness rides on bounded task-contracts + the spawn-fail-fall (Step 3), not on the platform forbidding a worker to spawn.**

> This contract is English so any model can read it; user-facing output follows the user's language (translate prose, technical terms verbatim — see **Always**).

## Step 0 — A valid ranking must exist first (the Lock)

Routing needs a tier ranking. Read the GLOBAL shared ranking `~/.claude/.coaltipple/ranking.json` (cheap file read — it is platform-level: built ONCE and shared across all your projects, since the model list does not vary by project). **Token-bound the freshness check: do NOT re-enumerate the live model list every session** (it costs tokens). Trust a cached ranking that is valid + younger than `rankingRefreshDays` (default 30): use it as-is. Re-enumerate the live list + rebuild ONLY when the cache is missing/corrupt/incomplete, OR older than `rankingRefreshDays`, OR you incidentally notice a model not in it (the user names a new one · a spawn picker offers an unranked option · an error cites an unknown model). A vendor list change (5 models to 10 overnight) does NOT break routing: it keys off the stable tier STRUCTURE (`local < low < mid < heavy < reasoning`), and any unfamiliar model classifies as `heavy` (never cheap) until the next rebuild — the exact list is the fragile layer, the tier structure is the robust one routing rides on.

**`rankingMode` (config) decides WHO builds the ranking:** `auto` (default) = YOU build it by introspection (the steps below) and overlay `modelTiers` pins — in auto mode accuracy is paramount: introspect carefully, never guess, an unfamiliar model → `heavy`, your own self-id is ground truth. `manual` = the HUMAN owns the ranking: take `modelTiers` as authoritative (+ the alias floor `haiku < sonnet < opus` for any tier they left unpinned) and do NOT introspect-override — use it when the user wants fully deterministic control. **If it is missing / stale (model list changed) / corrupt / incomplete → rebuild it on the spot:**
1. Work out which model tiers this platform can spawn (you know this — introspect yourself, don't trust a name table). Order them `local < low < mid < heavy < reasoning` by capability. **An unfamiliar/new model = `heavy` always, never cheap.**
   - **Usability = capability × availability × context-fit (important):** rank by capability **but route only to options actually available right now.**
     - **Availability ≠ catalog (the most important Step-0 distinction):** introspection / the model catalog determines ONLY the TIER STRUCTURE (which models exist + their capability order) — it does NOT prove a model is reachable. **NEVER treat a model as available because introspection/the catalog lists it:** a plan may disable it, it may be out of quota, or it may be gone. AVAILABILITY is discovered only at SPAWN-time — a spawn that errors instantly ("X is currently unavailable", 0-token) is the proof (proven live: a `fable` spawn returned unavailable instantly). So the ranking lists the best-KNOWN model per tier (the structure), and the spawn-fail-fall (Step 3 / Damage control) CORRECTS it at runtime. Rank by the catalog; never trust the catalog for reachability.
     - **disabled / quota / plan:** the strongest model but disabled (e.g. a flagship the vendor turned off), out of quota, or **not in the plan** = don't route to it (spawn fails) → that tier falls back to the next available option (e.g. reasoning drops to Opus @ max). Opus quota is limited → once hit, temporarily unavailable until reset.
     - **context variant (256k/1M) = a context-capacity axis ORTHOGONAL to capability** (1M > 256k > standard for large input) **+ plan-gated** (e.g. 1M = Max+ only). Pick the SMALLEST variant that fits *input + an output/reasoning margin* — don't over-provision (a 180k input fits a 256k variant; jumping to 1M just wastes cost/latency). Capacity (which variant) and capability (which tier) are SEPARATE axes: a big-input AND hard task wants strong-tier + big-variant (e.g. Opus-256k), never cheap-tier + big-variant. Don't hardcode window sizes (they vary by model + change over time) — reason from "does input + margin fit". **CAPACITY CEILING (variants RETIRE — never assume 1M exists): the largest AVAILABLE variant is the ceiling, discovered at SPAWN-time like availability (a 1M spawn that errors instantly = 1M gone; proven live — Opus 4.8 1M was retired). If input + margin EXCEEDS the ceiling, do NOT fall to a smaller-context cheaper tier (capacity ≠ capability — a lower tier has LESS context, the wrong direction) and do NOT silently overflow: CHUNK the input across passes/workers, OR keep it on the largest-available main / hand back.** Delegate-vs-keep for a big input → the context-floor in Step 2.
     - **auto-recover:** disabled→enabled or a plan change → the available set changes → the hash changes → re-classify yourself.
   - **If you are a low tier (e.g. main is Haiku):** the deterministic floor / alias is the **authority** — refine only what you are sure of, **don't override the floor by guessing**. Self-identity (which tier you are) is always trustworthy; ranking *other* models relies on the floor (same principle as grade: deterministic, not self-assessment). On CC the alias `haiku < sonnet < opus` is a constant — use it directly, no guessing.
2. Know your own tier (you know which model you are) + whether this platform lets you choose the worker's model/effort.
3. Write `{ schemaVer, listHash, complete:true, source:"introspection", tiers:{...} }` to `~/.claude/.coaltipple/ranking.json` (atomically).

**If a ranking genuinely cannot be built → turn routing OFF: just work normally.** The skill never *breaks* — it only *routes* or *routing is off*.

**Ranking robustness — maximal hardening at minimal token (the rule: make staleness SAFE, don't refresh FREQUENTLY — safe staleness is exactly what lets the check be rare):**
- **Self-heal from free signals, never poll:** a worker spawn that ERRORS (model gone / renamed / disabled / out of quota) → fall to the next tier now AND mark the ranking stale so the next routing rebuilds it — a failed spawn is free, accurate proof the list moved (cheaper + truer than any periodic poll). Combined with the cadence + incidental-notice triggers above, that is the COMPLETE check set; never enumerate the live list otherwise.
- **Bootstrap-upgrade (a seeded floor is not a real enumeration):** a ranking whose `source` is `"install-floor"` or `"heuristic-floor"` AND whose `listHash` is the empty-list hash is a never-introspected BOOTSTRAP — the installer/conductor seeded the alias floor without ever enumerating the live list. Its `complete:true` is self-refuting: the seeder attests "the floor is seeded", NOT "the live list was enumerated". So on the FIRST route by a capable main (`rankingMode:auto` + you are not a floor tier), UPGRADE it: introspect once and rewrite `source:"introspection"` + a real `listHash`. The signal is CHEAP — the `source` field alone says "bootstrap", no live enumeration needed to detect it (`isBootstrapRanking()` in classify.mjs is the deterministic check). This fires ONCE (after the upgrade `source` is no longer a floor source), so it does NOT re-enumerate every session — the token-floor holds. A bootstrap ranking is still VALID (routing never stalls on it): until the upgrade, it routes on the alias floor + unknown→heavy, exactly the safe-staleness everything else relies on.
- **Every staleness mode degrades SAFE (this is why rare checks are enough):** unknown/new model → `heavy` (never cheap) · listed-but-gone → spawn-fail → fall · unsure availability → try-then-fall · unsure tier → `heavy` (over-, never under-provision) · your own SELF-ID is always a correct anchor. Worst case of a stale ranking = a little over-provisioning or one failed-spawn-then-fall — NEVER a wrong-cheap route. Staleness being safe is precisely why you can afford to check rarely.
- **`modelTiers` pin beats introspection's blind spot:** a model released after your training is invisible to introspection (proven: a mid-tier main enumerated only the models in its own frozen system-context, missing newer ones). The user names it in `.coaltipple.json` `modelTiers: { <tier>: "<model>" }`; honor pins FIRST when building the ranking — they win over auto-classification (front of the tier). This is the one human-ground-truth override for what introspection literally cannot see.
- **Token floor:** an idle session pays NOTHING (the ranking is read only when you actually route) · a routing session pays a cheap file read + age check · the expensive enumerate+rebuild fires ONLY on the signals above. The check is rare-by-design, never per-session polling.

## Step 1 — Grade the task (deterministic, not self-assessment)

Grade 1–5 by the rubric — **size + sensitive-path + keyword**, not "it feels easy" (a cheap model is overconfident):

| Grade | Signal |
|---|---|
| 1 | read / search / format |
| 2 | ≤1 file, <500 lines, not sensitive |
| 3 | 2–3 files / 500–1000 lines |
| 4 | >3 files / ≥1000 lines / **sensitive path (auth, crypto, payments, migrations) — forces ≥4 even at 1 file** |
| 5 | concurrency / mutex / race / crypto-logic / timing-attack / proof — **forces 5 regardless of size** |

**Size = content size (general-purpose):** code counts lines/files · text/translation/research/docs count words/chars (same thresholds — "lines" is just the code instance of a universal content-size axis). **Difficulty (keyword/sensitive) always overrides size** — a tiny but hard task (1 line of crypto / a short legal-contract translation) = high grade.

**Keyword groups are config-tunable (`.coaltipple.json` `keywords`) — honor them when you grade:** the careful-keywords are grouped by task type (`concurrency`/`crypto`/`security`/`coding`, `math`, `knowledge`, `domain`, `creative`), each carrying a `grade` floor plus optional `sensitive` (never-down) and `preserveVoice` flags. Fold in BOTH the built-in groups AND the user's config overrides: a prompt matching a group's words floors the grade to that group's `grade` and inherits its flags. This is the user's lever to tune per-domain caution (add/remove a word, change a grade per group); a group they omit keeps the built-in. The conductor's 0-token hint reflects only the built-ins, so YOUR read of the config is the authority. **Match a group by the task's INTENT in ANY language** — a bug-scan, audit, or security-review request matches the `audit` group by MEANING, not by a literal English word (the skill ships worldwide; grade the meaning, never require English in the prompt — the English `words` are only examples + the 0-token conductor hint).

**Read WHY the grade is high — it sets the DIRECTION you route (the grader's `reasons` tell you which):**
- **High by DIFFICULTY** (sensitive path / hot keyword) → escalate-**UP**: you need a stronger tier for correctness.
- **High by SIZE alone** (`reasons` show only `size: N files/units`, no sensitive/keyword — a 40-file rename, 48 CRUD handlers, bulk codegen/translation) → this is a high-**EFFORT** signal, NOT a high-tier one → **delegate-DOWN to a cheap tier + high effort.** The size threshold inflates the grade to protect you from under-routing big *complex* work; once you can see the work is MECHANICAL, route it DOWN. Never escalate-up or climb-the-staircase on bulk-mechanical work just because its breadth bumped the grade — bulk is the EFFORT knob, difficulty is the TIER knob. (Observed: mains graded a 48-handler boilerplate scaffold "4" and one then sat on the staircase from mid instead of delegating down — a size-grade misread as a tier signal.)
  - **EXCEPTION — a whole-repo audit / bug-scan / security-review / code-review is high-by-DIFFICULTY, NOT size:** it spans many files (looks size-driven) but finding REAL issues needs CAPABILITY — a cheap tier returns a confident shallow all-clear. Route it UP, or keep it on a capable main; NEVER delegate-down-to-cheap, and if YOU are the floor (Haiku) NEVER self — escalate UP. (Observed: a Haiku main, told "scan for bugs", self-ran it and reported a shallow "no bugs" — the overconfidence trap. Bug-finding is the TIER knob, not the EFFORT knob.)

If the conductor injected a hint, respect it.

**qualityBar (config 0–100) — the acceptable-quality bar:** grade picks the *starting* tier (Step 2); `qualityBar` is the *line the result must clear*, or routing climbs the model ladder (Step 3). Default 60.
- **Plain meaning** (not an opaque number): "accept work of quality ≥ qualityBar%". Set it to your own standard — `90` = critical, must be exact · `60` = competent/serviceable (default) · `40` = cheap-and-cheerful.
- **0** = anything passes → stay at the starting tier (the cheapest that can do it), never climb. **100** = almost nothing passes below the top → climb to the top (except a task the lower tier already does perfectly, e.g. trivial → no climb = save where quality isn't at risk). The extremes emerge from the climb mechanism — nothing is hardcoded.

## Step 2 — Route (decide)

**HARD GATE, before grade or anything else — sensitive = NEVER delegate-DOWN, and this does NOT depend on your grade (the single most important safety rule):** if the task touches anything sensitive/security — crypto, timing-attack, constant-time, auth, authorization, payment, secret, token, password, session, security (the grader + conductor flag these, plus any `.coaltipple.json` `keywords` group you marked `sensitive`) — then **delegate-DOWN is FORBIDDEN, whatever number you graded it.** Do NOT gate this on grade ≥4/5: a weaker main UNDER-grades sensitive work and slips straight past a grade-gated rule. (Observed live: a main graded a hand-rolled constant-time compare *medium*, then delegated it DOWN to a cheaper tier — "known pattern, save tokens". The under-grade bypassed the safety rule and shipped hand-rolled crypto to a weaker model — the exact failure this gate exists to stop.) **The keyword is the gate, not your grade.** For sensitive work: escalate UP, keep it on main, or use a vetted built-in (`crypto.timingSafeEqual`) — never delegate-down, never hand-roll security on a cheaper tier. "Cheaper tier to save tokens" on security work = false economy: a plausible-but-wrong timing leak PASSES your tests. **This holds under a QUOTA / limit-hit too:** if the safe tier is quota-blocked, a sensitive task WAITS for reset, stays on a capable main, or hands back — it does NOT fall to a cheaper tier just because the limit was hit (the limit-hit fallback floors sensitive work at its safe-minimum tier — see Damage control).

**Part of this gate — the keyword flags are ENGLISH-ONLY, so on a NON-ENGLISH task YOU are the sensitive-gate authority:** the deterministic grader + the conductor's hint match English literals, so a Thai/CJK/Arabic/Cyrillic prompt meaning "scan for bugs" / "constant-time compare" / "auth bypass" matches NOTHING — no keyword fires, the grade reads trivial, and a grade-gated reading of this rule would let the never-down backstop silently vanish. Do NOT rely on the keyword flag for a non-English prompt: **apply the never-down gate by the task's MEANING / intent** (a non-Latin prompt about crypto/auth/payment/security is sensitive even though no flag fired). The conductor injects a generic "non-English -> grade by meaning" nudge, but the authority is your own read of the intent, not the (absent) flag. This is the hard gate, not a Step-1 aside: the gate fires on MEANING, and the keyword is only its English fast-path.

**`mode` (config, default `auto`) sets the DIRECTIONS you may route — honor it (the sensitive HARD GATE above OVERRIDES it):**
- `auto` (default) = route BOTH directions per grade (delegate-down for large+cheap, escalate-up for hard).
- `delegation` = the budget mode — delegate-DOWN is allowed, escalate-UP is SUPPRESSED (when a result misses the bar, do it yourself / hand back rather than climbing the tier).
- `escalation` = the quality mode — escalate-UP is allowed, delegate-DOWN is SUPPRESSED (never push work down to save tokens; keep it on main or climb).
- `off` = routing OFF — do the task yourself (the conductor also short-circuits its forcer on `mode:"off"`).
- **Override:** a SENSITIVE task still obeys the hard gate under ANY mode — never-down always holds, and escalate-UP for sensitive work is always permitted even in `delegation` mode (safety beats the budget knob).

**Per-domain `disableRouting` (config) — routing OFF for a matched domain:** if the graded task's domain (`coding` / `text` / `math` / `research`) is listed in `disableRouting`, OR the list includes `'all'`, then routing is OFF for THIS task — do it yourself (no delegate-down, no escalate-up). Map the task to its domain by its nature (code work → `coding`; translation/docs/prose → `text`; proofs/derivations → `math`; literature/citation/claim work → `research`). (The conductor only short-circuits on `'all'`; per-domain matching needs the graded domain, so it is YOUR check here.)

Two knobs: **TIER** (coarse, burns Opus quota) × **EFFORT** (fine, cheap, low→max). **Escalate cheapest-first, EFFORT then VERSION then TIER:** raise EFFORT (same model, think/iterate harder), then raise VERSION (a stronger model of the SAME tier, e.g. Opus 4.6 -> 4.8, cheaper and closer than a tier jump), then raise TIER (the next tier up). Exhaust each lever before the next; never skip to a higher TIER while a stronger version of your current tier is untried (think harder on Sonnet before jumping to Opus — Opus quota is scarce; once hit it is hard-blocked).

| Grade | Move | Tier · Effort |
|---|---|---|
| 1–2 | **delegate-down** *(only if the task is large — see floor)* else do it yourself | low · low–med |
| 3 | do it yourself or delegate | mid · med→**max** before raising tier |
| 4 | escalate-up *(if difficulty-driven)* / delegate-down *(if size-driven + mechanical; but if YOU are the floor tier this is SELF, no "down" exists)* | heavy · effort by SIZE · sensitive → never down |
| 5 | escalate-up / do it yourself | strongest · effort by SIZE (small = low, deep/large = max) |

- **The tier column = the CEILING, not the START — climb the staircase to it:** escalate ONE rung at a time and verify each (Step 3); a lower rung often clears the bar (cheap tiers are more capable than they look — proven this session). Reach the ceiling tier only when a lower rung FAILS verify; jump straight to the top on far-below, out-of-attempts (`maxTotalAttempts`), OR when the failure mode is subtle / un-verifiable — crypto (timing leaks), numerical precision, security correctness — where a cheaper tier's plausible-but-wrong output would PASS your check. **Rule: climb when verify is your safety net; JUMP to the reliable tier when verify can't catch the failure.** Don't pre-jump to the top just because the grade is high — that over-provisions TIER the same way auto-xhigh over-provisions effort. (A sensitive/hard grade still forces escalation OFF the current cheap tier — climb at least one rung, just don't leap to the ceiling. Delegating sensitive work DOWN is already forbidden by the hard gate above.)
- **EFFORT tracks OUTPUT-SIZE/iteration, NOT difficulty — the two knobs are independent:** TIER is the correctness lever (difficulty/sensitivity → which model); EFFORT is the size lever (how much to produce/iterate → low→max). A hard task with SMALL output = strong TIER + LOW/moderate effort: a 10-line constant-time compare gets the reasoning tier for *safety* but LOW effort — it's 10 lines; do NOT burn xhigh/max on it. Reserve xhigh/max for genuinely LARGE or deep-iteration work (a formal proof, a big refactor). Burning xhigh on a tiny fn because it's *hard* is the over-provisioning trap — the TIER already bought the correctness; effort only sizes the work. (Observed: mains put xhigh on the 10-line crypto fn — difficulty leaking into the effort knob.)
- **Effort rubric (by OUTPUT SIZE, not difficulty):** `low` = a lookup / <=30 lines / 1-2 sentences · `medium` = a function + tests / a short proof / a paragraph · `high` = a multi-case proof / a multi-file change / a long document · `max` = a formal proof with all cases / a big refactor / an exhaustive sweep. Difficulty + sensitivity already chose the TIER; this only sizes the work.
- **Effort is the ALWAYS-ON lever (it pays off even when the TIER cannot move):** at the top tier, or when only one tier is available (cheaper tiers off, or the user runs a single model), you cannot escalate the TIER, but you STILL optimize by scaling EFFORT per task (low for the easy one, high for the hard one). Most routers only pick a tier (worthless with one tier); CoalTipple's effort lever delivers savings at ANY fixed tier. A self-executing top main (reasoning tier off) IS still routing: it pins the tier and tunes effort per task.
- **Delegate-down FLOOR (most important):** spawning a worker has a high fixed overhead (~tens of thousands of tokens — measured). **Delegate down only for a task big enough that doing it yourself would cost far more than the overhead** (≥ `delegateMinLines`, default 120). Small-to-medium tasks → **do them yourself**; offloading loses tokens. `delegateMinLines` is a CODE-line proxy for this break-even — cheap-per-line work (translation, formatting, plain text) burns far fewer tokens per line, so it needs proportionally MORE than its raw line count before delegating pays off. Judge the real effort against the overhead, not the line count.
- **Budget gate before you fan out (the proactive limit-hit guard) · spawning is the single biggest spend you control:** a worker pays off only when the task clears the floor above AND there is clear BUDGET HEADROOM for the overhead. Near a session/quota limit, or right after a limit-hit, the cheapest correct route is often NO worker: collapse to inline-self. Above all, do NOT fan out SEVERAL workers at once for marginal-value, context-light, mechanical work (a multi-file grep/compare, a small audit, a consistency sweep) the main can do inline: N spawns x ~tens-of-thousands overhead near a budget edge IS the limit-hit, and a worker that dies on the limit returns NOTHING (partial findings unrecoverable, unlike a journaled file-mutation). `resolveWorker` (Damage control) is the REACTIVE fall once a spawn is already blocked; THIS is the proactive gate before you spawn at all. (Observed live: a main fanned out 2 audit subagents for sub-floor mechanical work, hit the real session limit, and lost both to 0-token deaths.)
- **If YOU are the floor tier (the cheapest — e.g. main is Haiku):** no tier exists below you, so delegate-DOWN is off the table. A big size-driven mechanical task (the one the grade table routes "delegate-down" for a higher main) has NO down-target at the floor: it collapses to SELF. Do it yourself and CALL it self; a "delegate-down to myself" label is the self-contradiction that then slips the mechanical work UP to a costlier tier. Cheap/mechanical work you can do → **do it yourself** (delegating it sideways or UP to a costlier tier wastes tokens for zero quality gain). Work beyond your competence (hard / sensitive / too-big-context) → **escalate UP**: do NOT keep it on yourself just because "it isn't sensitive" or because you hold the context — a hard task left on the floor tier is under-powered. (Observed: a Haiku main delegated boilerplate UP to mid, and kept a 180k refactor on its own low tier — both backwards. At the floor the rule is simply: self-for-cheap, up-for-hard.)
- **Context floor (the worker starts FRESH) — distinguish two kinds of "big context":**
  - **Accumulated CONVERSATION context** (files you read, decisions made, the user's intent, the back-and-forth so far) — NON-transferable; a fresh worker can't see it. Work whose correctness depends on it → **keep on main** (compressing it into a contract is lossy + expensive). This is the real context-floor.
  - **On-disk / re-readable INPUT** (a codebase, a large document, logs) — a worker CAN read it itself from a path in the contract. A large *on-disk* input is NOT a reason to keep work on main — it's a reason to pick a worker with a big-enough context VARIANT (Step 0): choose its TIER by difficulty and its VARIANT by input size, independently.
  - **Worked example ("refactor a 180k-token codebase in one pass"):** the 180k is on-disk → re-readable → delegatable. Route by difficulty (a hard refactor → heavy tier) + capacity (180k → 256k variant) = an Opus-256k worker reading the files. Do NOT collapse it into "big input → keep it on my own (cheap) tier" — that sacrifices capability for a capacity problem a variant solves. (Exception: if you ALREADY hold the input in a large-context main, keeping it there is free — no spawn overhead.)
- **ULTRACODE** (xhigh + multi-agent fan-out) = the top rung, for genuinely *broad + independent* work only; workers run a cheap tier, you orchestrate (cap concurrent workers at `maxConcurrentSubagents`, default 4 — they share one rate limit). **fast-mode is not a routing choice** — attach it only on an explicit human latency request, and only if `fastModeOnLatencyRequest` (default true); never as a routing rung.
- **Voice:** when `preserveVoiceForUserFacing` (default true), never delegate the final user-facing deliverable to a cheaper model — this INCLUDES a translation, summary, or research write-up (the prose IS the deliverable), even when it is bulky. For a large user-facing translation/document, doing it yourself usually beats delegate-then-review, because reviewing it to protect voice + terminology costs about as much as redoing it. (Bulk MECHANICAL work — renames, formatting, codegen — is not "deliverable prose" and stays delegate-able.)

## Step 3 — If you delegate

1. **Task contract:** a compact task contract is required when `requireTaskContract` (default true) — give the worker goal + constraints + interface + done-criteria. A thin brief makes a cheap worker build the wrong thing with confidence (proven). Keep it compact.
2. Spawn the worker at the chosen tier/effort — **lead the spawn's description/label with the chosen model + effort so the human sees exactly what is running** (e.g. `[haiku·low] scaffold 48 handlers`, `[opus·max] formal proof`, `[sonnet·high] refactor auth`; Claude Code renders that description as the task chip). Require it to return a **compact diff/summary**, not whole files. **The worker MUST be BOUNDED — never spawn an open-ended worker that can loop:** the task contract's done-criteria (item 1) is what makes it terminate (a worker with no clear "done" — a research-style / "keep improving" brief — can run forever and burn the budget). `subagentTimeoutSeconds` (default 150) catches a STALL (Damage control); the done-criteria prevents the LOOP. A bounded worker returns its result and terminates.
3. **Spawn-fail-fall — the never-fail availability loop (drives `resolveWorker`):** a spawn can ERROR instantly (0-token) because the chosen model is **unavailable / disabled / out of quota / gone** — e.g. "X is currently unavailable" (proven live: a `fable` spawn). Availability is knowable ONLY here, at spawn-time (Step 0). When it errors: **(a)** add that model to a `blocked` set; **(b)** call `resolveWorker(ranking, desiredTier, {blocked, floorTier})` (classify.mjs — walks `desiredTier`→`floorTier`, skipping `blocked`); **(c)** spawn the returned next-available worker; **(d)** repeat (a)–(c) on each fresh unavailable error; **(e)** if `resolveWorker` returns `null` (everything is blocked down to the floor), STOP — for a SENSITIVE task the `floorTier` is its safe-minimum, so `null` means hand back / wait for reset and **NEVER breach never-down** by falling cheaper. Mark the ranking stale (a failed spawn is free proof the list moved — Step 0). **Guarantee: routing reaches a working model, OR hands back cleanly — it never gets stuck on an unavailable model, and never falls sensitive work below its floor to escape a block.** (This is the DOWN-for-availability move; the quality climb below is the UP-for-quality move — opposite directions, do not conflate.)
4. **Verify against qualityBar:** verify rigor follows `qaOnMerge` (strict / standard / off; default standard) — `off` skips the merge-verify (trust mode), `strict` = exhaustive. The output may "look right but be wrong" (proven) — **never trust it blindly, and never let the worker grade itself.** Measure against external criteria = the contract done-criteria + a **domain-appropriate objective check** (code: build/test/lint/type-check · text/translation: every segment present + consistent + format preserved · research: every claim has a real, authoritative source, verified with the web/doc tools *you have yourself* — do not depend on another skill · math: check the steps / substitute back). **RUN the check — actually execute the tests/build/parse; do NOT eyeball the code and reason about whether it looks right. A cheaper main especially defaults to eyeballing: it catches obvious misses but mis-judges subtle bugs (live test — low and mid mains both read a buggy `1.5h` as 5400/3600 by eye; only running it revealed the true 18000, a silent wrong number). Eyeballing is not verifying.** Clears the bar → done.
   - **Below the bar → climb the staircase:** near the bar + attempts remain → **escalate one rung, where one rung = the next lever in the effort/version/tier order:** a stronger VERSION of your current tier first (e.g. Opus 4.6 -> the best Opus 4.8), then the next TIER; don't retry the same model, and don't skip to a higher tier while a stronger same-tier version is untried. **Far below the bar (the next rung won't clear it), or out of `maxTotalAttempts` (default 2) → jump straight to the top tier** (don't die by a thousand cuts). At the top and still failing → hand back to the user directly.
   - **competence-introspection:** if you *know* from the start that the task exceeds your ceiling (the CT definition: "I cannot do this") → escalate immediately, don't waste an attempt trying first.
5. **Never let a cheap main QA an expert's work** — if you escalated *up*, let the strong worker own both the hard task and the integration.

## Memory anchor (the context-floor mitigation)

A worker starts context-FRESH (the context-floor in Step 2). A **memory anchor** — the project's own memory/conventions file — gives a fresh worker (and a resuming main) context BEYOND the bare task contract, so delegation doesn't drop project knowledge.
- **Find the anchor:** if `contextFiles` is set, those file(s) ARE the anchor — point the worker contract at them (read for conventions/context). If `contextFiles` is empty, rely on the platform's OWN memory: it auto-loads `CLAUDE.md` / `AGENTS.md` into sessions, so no action is needed.
- **Lazy offer (once, never spam):** when you are about to delegate context-dependent work AND no anchor exists (`contextFiles` empty AND the project has no `CLAUDE.md`/`AGENTS.md` the worker would inherit) AND `memoryOffer` is `auto` AND you have not yet offered this session → offer via your question tool, exactly 3 options: **Create** a memory file · **Skip** · **Choose** an existing file. At most ONCE per session.
  - **Create** → create a memory file (the platform convention, e.g. `MEMORY.md`) + seed a concise handover snapshot (key conventions + decisions + current state) + set `contextFiles` to it.
  - **Choose** → set `contextFiles` to the user's named file (any name) + load it.
  - **Skip** → set `memoryOffer` to `off` (persist — never re-ask) + proceed best-effort (the task contract + `state.json` only).
- **Write-policy (respect the user's file):** a MISSING anchor → create + seed once. An EXISTING anchor → load it + append/crystallize, but **never clobber** it (the same discipline as config-preservation).
- **Maintain:** while an anchor is active, keep it updated + crystallized as the session progresses — append key decisions/conventions, consolidate duplicates, refine; never duplicate what the repo already records.
- **No spam, never locked:** CT asks at most once, then persists the choice; the user changes it anytime via `/coaltipple memory` — CT never re-prompts on its own.

## Damage control (you can't control the limit · you can control the damage)

- Don't fight the platform's rate-limit retries (the platform handles 429 itself). Your job = bound the blast radius of a mid-run death.
- **Silent stall:** a background worker that returns nothing past `subagentTimeoutSeconds` (default 150) → treat it as failed and re-route (the platform may not signal a silent stall).
- **Limit-hit / unavailable fallback (a tier goes unreachable mid-route — quota exhausted / disabled / not in the plan / a spawn that errored "model unavailable" / a model gone):** resolve the next worker by walking DOWN the ranking to the next AVAILABLE tier — the availability move, the OPPOSITE of the quality climb (Step 3). Within-tier first (a blocked Opus 4.8 → Opus 4.7), then drop a tier (all Opus quota-blocked → Sonnet). **BUT never below the task's floor:** a SENSITIVE task floors at its safe-minimum tier — if nothing is available at/above it, hand back or wait for reset, NEVER fall crypto/auth/payment to a cheaper tier on a quota-hit. Everything blocked → do it yourself if you can clear the bar, else hand back honestly. (`resolveWorker` is the deterministic substrate; the spawn-fail-fall loop that drives it is Step 3 item 3 — this is the SAME loop, whether the block hits on the first spawn or mid-route.)
- Have the worker write its proposal to `.claude/.coaltipple/proposed/`; you merge it deliberately → a mid-run death never corrupts real files. Journal finished subtasks to `.claude/.coaltipple/state.json` → resume skips the done ones.
- **For a file-mutating delegation in a git repo, prefer worktree-isolation** (the worker edits an isolated copy; a mid-run death discards it) — else the `.claude/.coaltipple/proposed/` sandbox is the git-agnostic fallback.
- **A step with a side-effect (bash mutation, external call, commit) = never delegate + never retry** (retry = doing it twice; bash isn't checkpointed). A git user → a commit is a real recovery line, used as an extra recovery boundary per `gitRecoveryBoundary` (auto / on / off; default auto — `auto` = use it only inside a git repo) — but never assume git exists.

## Always

Honor the MERGED config (every value is tunable): the global `~/.claude/.coaltipple.json` (your defaults for all projects) overlaid by the project `.claude/.coaltipple.json` (per-project overrides — project value wins per key). A project file is OPTIONAL: absent = the global config plus schema defaults apply. Consent-gate token spend — never fan out costly work silently. **Respond in the user's language** (auto-detect from their messages, or the `language` config): translate the prose, but **never translate technical terms** — commands, paths, identifiers, tier/model names, severity labels, and config keys stay verbatim. `/coaltipple off` turns routing off; `/coaltipple stats` shows approximate savings (an estimate — there is no cost API); `/coaltipple memory [on|off|set <file>]` sets up or disables the memory anchor (on = offer/setup · off = disable + persist · set = name your file).

## Self error-report

If CoalTipple itself misbehaves — a contradictory instruction, a procedure that loops or dead-ends, a routing decision that is clearly wrong, the Lock refusing a valid ranking, or a config key that does not behave as documented — STOP and surface it: give the user a one-paragraph summary of what went wrong and where. OFFER to file it at `github.com/TheColliery/CoalTipple/issues`. Never auto-submit; never include unapproved code or paths. A skill that hides its own failures cannot be fixed.

Honest blind spot: this report is MODEL-driven, so it fires ONLY for misbehavior the model NOTICES. It cannot see a fail-silent hook death (Phoenix hooks exit 0 on error by design), a plausible-but-wrong worker output that slipped past a weak check, or a missed delegation. A clean run means "nothing noticed", not "nothing wrong".
